# 硬件调试升级排查协议

## 概述

同一类问题连续失败 3 次，**停止重试，与用户一起排查根因。**
3 次失败说明存在系统性问题，继续尝试只会浪费时间。

## 三次升级规则

**核心原则：** 每次失败必须收集新的诊断证据。相同的动作不做第二次。

### 调试器连接

| 次数 | 动作 | 收集的证据 |
|------|------|-----------|
| 第 1 次 | `disconnect` + 重新 `connect` | 是否有旧会话残留 |
| 第 2 次 | `list_probes` 检查 USB | 调试器是否可见 |
| 第 3 次 — 升级 | 停止，检查硬件 | USB 线缆、目标板供电、SWD 接线 |

### 烧录编程

| 次数 | 动作 | 收集的证据 |
|------|------|-----------|
| 第 1 次 | 重试一次烧录 | 是偶发还是持续失败 |
| 第 2 次 | `flash_erase(all)` + 重试 | Flash 是否被写保护或损坏 |
| 第 3 次 — 升级 | 停止，检查文件和芯片 | ELF 文件完整性、Flash 容量、写保护 |

### RTT 附加

| 次数 | 动作 | 收集的证据 |
|------|------|-----------|
| 第 1 次 | 重试同一 CB 地址 | 是否时机问题 |
| 第 2 次 | `read_memory` 验证 CB 内容 | CB 结构是否正确 |
| 第 3 次 — 升级 | 停止，检查编译配置 | cfg.h 中 DBG_MODE、RTT 源文件是否加入编译 |

### RTT 无数据

| 次数 | 动作 | 收集的证据 |
|------|------|-----------|
| 第 1 次 | `get_status` 检查目标 | 目标是否 running |
| 第 2 次 | `read_memory` 检查 RTT 结构 | CB 结构是否被破坏 |
| 第 3 次 — 升级 | 停止，检查固件逻辑 | RTT 初始化顺序、栈溢出、HardFault |

## 升级报告模板

向用户报告时包含以下内容：

```
### 诊断报告：[失败类别]

**已收集证据：**
1. [第 1 次尝试结果 + 证据]
2. [第 2 次尝试结果 + 证据]

**已排除的原因：**
- [已确认正常的环节]

**疑似根因：**
- [基于证据的判断]

**建议下一步：**
- [具体的排查方向]
```

## 架构质疑

当硬件连接和烧录操作均确认正常，但固件行为经过 3 次修改仍达不到预期效果时：

**停止修改代码，重新审视整体方案。**

### 触发条件

1. 硬件连接正常（`list_probes` / `get_status` 通过）
2. 烧录成功
3. RTT 通信正常
4. 固件行为不符合预期
5. 已尝试修改代码 3 次

### 架构问题信号

| 信号 | 含义 |
|------|------|
| 每次修改修复一个现象，但暴露另一个新问题 | 模块间耦合过深，需要重新设计交互 |
| 修改需要"大范围改动"才能实现 | 当前架构不支持该功能，需要调整架构 |
| 同一个问题在不同条件下反复出现 | 根因在架构层而非代码层 |
| 修复看似简单但引入回归 | 接口契约不清晰，需要重新定义边界 |

### 行动

1. **确认前提** — 重新验证硬件、烧录、RTT 都正常
2. **质疑架构** — 当前方案是否在正确的方向上？
3. **与用户讨论** — 提出已收集的证据和架构疑虑，共同决定是继续修改还是重新设计方案
