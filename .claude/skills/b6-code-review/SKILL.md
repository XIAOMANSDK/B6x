# B6 Code Review

B6x SDK 代码审查工具，检查 MISRA C 合规性、嵌入式最佳实践、BLE 规范和安全性问题。

## 触发

`/b6-code-review <file_or_directory>`

**参数**:
- `file_or_directory`: 要审查的文件路径或目录
  - 单文件: `/b6-code-review projects/bleHid/src/main.c`
  - 目录: `/b6-code-review projects/bleHid/src/`
  - 项目: `/b6-code-review projects/bleHid`

## 审查类别

### 1. MISRA C 合规性 (Critical)

| ID | 检查项 | 规则 | 严重性 |
|----|--------|------|--------|
| M01 | 固定宽度类型 | 必须使用 `uint8_t`/`int16_t` 等，禁止 `int`/`unsigned` | 🔴 Critical |
| M02 | 变量初始化 | 所有局部变量必须显式初始化 | 🔴 Critical |
| M03 | const 修饰 | 只读指针参数必须用 `const` 修饰 | 🟡 Warning |
| M04 | static 限制 | 仅内部使用的函数/变量必须用 `static` | 🟡 Warning |
| M05 | 函数参数数量 | 参数 ≤ 4 个，超过需用结构体封装 | 🟡 Warning |
| M06 | 禁止动态内存 | 禁止 `malloc`/`free`/`calloc`/`realloc` | 🔴 Critical |
| M07 | 禁止递归 | 禁止递归函数调用 | 🔴 Critical |
| M08 | 禁止 goto | 禁止 `goto` 语句 | 🟡 Warning |
| M09 | 显式类型转换 | 禁止隐式类型转换，必须显式转换 | 🟡 Warning |
| M10 | 返回值检查 | SDK 函数返回值必须检查 | 🔴 Critical |

### 2. 嵌入式专用检查 (Critical)

| ID | 检查项 | 规则 | 严重性 |
|----|--------|------|--------|
| E01 | ISR 简短性 | 中断服务程序 ≤ 50 行 | 🟡 Warning |
| E02 | ISR 非阻塞 | ISR 内禁止 `while` 等待、延时调用 | 🔴 Critical |
| E03 | volatile 使用 | ISR 与主程序共享变量必须用 `volatile` | 🔴 Critical |
| E04 | 寄存器访问 | 硬件寄存器指针必须用 `volatile` | 🔴 Critical |
| E05 | 位操作宏 | 使用 `BIT_SET`/`BIT_CLEAR`/`BIT_TOGGLE` | 🟡 Warning |
| E06 | 空闲等待 | 空闲循环应调用 `__WFI()` 或 `__WFE()` | 💡 Info |
| E07 | 大数组栈分配 | 局部数组 > 256 字节应使用 `static` 或全局 | 🟡 Warning |
| E08 | 未使用变量 | 未使用的变量应移除 | 💡 Info |

### 3. BLE 规范检查 (Warning)

| ID | 检查项 | 规则 | 严重性 |
|----|--------|------|--------|
| B01 | UUID 备案 | 自定义 UUID 必须在 MEMORY.md 记录 | 🟡 Warning |
| B02 | 连接参数 | Interval 7.5-100ms, Latency 0-4, Timeout 1000-6000ms | 🟡 Warning |
| B03 | Profile 完整性 | 服务必须有 init/read/write/notify 回调 | 🟡 Warning |
| B04 | BLE 宏定义 | 使用 `BLE_NB_SLAVE` 而非硬编码连接数 | 💡 Info |
| B05 | 事件处理 | 所有 BLE 事件必须有处理分支 | 🟡 Warning |

### 4. 代码风格检查 (Info)

| ID | 检查项 | 规则 | 严重性 |
|----|--------|------|--------|
| S01 | 命名规范 | 文件小写下划线，函数模块_功能 | 💡 Info |
| S02 | 全局变量前缀 | 全局变量使用 `g_` 前缀 | 💡 Info |
| S03 | 宏命名 | 宏使用大写+下划线 | 💡 Info |
| S04 | 缩进 | 4 空格，无 Tab | 💡 Info |
| S05 | 行宽 | ≤ 100 字符 | 💡 Info |
| S06 | Doxygen 注释 | 函数应有 `@brief` `@param` `@return` | 💡 Info |
| S07 | 文件头注释 | 文件应有 `@file` `@brief` | 💡 Info |

### 5. 安全性检查 (Critical)

| ID | 检查项 | 规则 | 严重性 |
|----|--------|------|--------|
| C01 | 数组边界 | 数组访问必须有边界检查 | 🔴 Critical |
| C02 | 空指针检查 | 指针使用前必须检查非空 | 🔴 Critical |
| C03 | 整数溢出 | 算术运算需检查溢出 | 🟡 Warning |
| C04 | memcpy 长度 | `memcpy` 目标缓冲区必须足够大 | 🔴 Critical |
| C05 | 格式化字符串 | 禁止用户输入作为格式化字符串 | 🔴 Critical |

## 严重性级别

| 级别 | 标识 | 说明 | 处理 |
|------|------|------|------|
| 🔴 Critical | `CRITICAL` | 必须修复，可能导致系统崩溃或安全问题 | 阻断 |
| 🟡 Warning | `WARNING` | 建议修复，影响代码质量或可维护性 | 警告 |
| 💡 Info | `INFO` | 可选修复，代码风格建议 | 提示 |

## 执行步骤

### Step 1: 定位目标

1. 确定审查范围：单文件还是目录
2. 使用 Glob 获取所有 `.c` 和 `.h` 文件列表
3. 按文件大小排序，优先审查核心文件

### Step 2: 逐文件扫描

对每个文件：

1. 使用 Read 读取完整内容
2. 记录文件路径和行数
3. 识别文件类型（驱动/应用/BLE Profile/配置）

### Step 3: 分类检查

按优先级顺序检查，Critical 问题立即记录：

| 顺序 | 类别 | 关注点 |
|------|------|--------|
| 1 | 安全性 (C01-C05) | 可能导致系统崩溃的问题 |
| 2 | MISRA C (M01-M10) | 核心合规性 |
| 3 | 嵌入式 (E01-E08) | 硬件相关问题 |
| 4 | BLE (B01-B05) | 协议栈相关 |
| 5 | 风格 (S01-S07) | 代码质量 |

### Step 4: 使用 MCP 工具验证

遇到 SDK API 调用时：

1. 使用 `search_sdk(api_name)` 查询正确用法
2. 使用 `inspect_node(node_id)` 确认参数要求
3. 对比代码实现与 API 规范

### Step 5: 生成报告

按输出格式汇总所有问题，按严重性排序。

## 检查模式

### MISRA C 模式

```regex
# M01: 固定宽度类型
\b(int|unsigned|long|short|char)\b\s+\w+\s*[;=]

# M02: 变量初始化
\b(uint\d+_t|int\d+_t)\b\s+\w+\s*;

# M06: 动态内存
\b(malloc|free|calloc|realloc)\s*\(

# M07: 递归 (函数名与调用匹配)
\b(\w+)\s*\([^)]*\)[^;]*\{[^}]*\b\1\s*\(

# M08: goto
\bgoto\b

# M10: 返回值未检查
=\s*\w+_init\(|=\s*\w+_config\(|=\s*\w+_enable\(
```

### 嵌入式模式

```regex
# E02: ISR 内阻塞调用
void\s+\w+_IRQHandler\s*\([^)]*\)[^}]*while\s*\(

# E03: 缺少 volatile (ISR 相关)
volatile\s+\w+\s+g_\w+

# E05: 直接位操作 (应使用宏)
\w+\s*\|=\s*\(1\s*<<|=\s*\w+\s*&\s*~\(1\s*<<

# E07: 大数组栈分配
uint\d+_t\s+\w+\s*\[\s*[0-9]{3,}\s*\]
```

### BLE 模式

```regex
# B01: UUID 定义 (检查是否备案)
0x[0-9A-Fa-f]{4}.*UUID|UUID.*0x[0-9A-Fa-f]{4}

# B02: 连接参数
CONN_INTERVAL|SLAVE_LATENCY|SUP_TIMEOUT

# B05: 事件处理
case\s+BLE_\w+:
```

### 安全模式

```regex
# C01: 数组访问 (需人工确认)
\w+\s*\[\s*\w+\s*\]

# C02: 指针使用 (需人工确认)
\*\s*\w+\s*=|\w+\s*->\s*\w+

# C04: memcpy
memcpy\s*\(\s*\w+\s*,\s*\w+\s*,\s*\w+\s*\)
```

## 注意事项

1. **误报处理**: 部分检查可能产生误报，需人工确认
2. **上下文理解**: 复杂逻辑需结合代码上下文判断
3. **增量审查**: 可仅审查修改的文件
4. **CI 集成**: 可在提交前自动运行，阻断 Critical 问题
5. **MCP 辅助**: 遇到不确定的 API 用法时，使用 MCP 工具验证
